# 노무관리 Pro v5 — 운영 모니터링 사용 매뉴얼

> 시스템 이상 징후를 운영자가 직접 확인하는 절차서.
> 외부 모니터링 서비스(Sentry 등) 미사용 — 모든 로그는 우리 Supabase에 저장.

---

## 0. 무엇을 보는 도구인가

관리자 페이지의 **[📊 모니터링]** 탭은 다음을 보여줍니다:
- 사용자 브라우저에서 발생한 에러 (사용자가 신고하지 않아도 자동 수집)
- 서버 함수 에러 / 504 / 가드 트리거
- 21중 데이터 유실 방지 가드의 동작 신호 (가장 중요)

**예: 이런 상황을 즉시 인지할 수 있음**
- "어제부터 50명에게 같은 화면에서 같은 에러 발생" → 신규 버그 조기 인지
- "이번 주 504가 평소 3배" → 큰 고객사 진입했거나 데이터 누적이 임계점
- "데이터 wipe 시도 가드가 차단" → 즉시 사고 가능성 점검

## 1. 권장 점검 주기

| 주기 | 무엇을 볼지 |
|------|-----------|
| **매일 1분** | 사이드바 [📊 모니터링] 메뉴의 빨간 뱃지 숫자 확인. 0이면 OK |
| **주 1회 5분** | [📊 모니터링] 탭 진입 → 통계 카드 + 상세 목록 훑어보기 |
| **사고 의심 시** | 즉시 진입 → 해당 시간대 로그 검색 |

---

## 2. 화면 구성

### 통계 카드 (상단 4개)
- 🔴 **error**: 진짜 에러 (코드 버그, 잡히지 않은 예외)
- 🟡 **warn**: 경고 (504 발생, 저장 일시 실패 등)
- 🛡️ **guard**: 가드 트리거 (데이터 유실 시도가 막힌 경우 — **가장 중요한 신호**)
- 📊 **총합**: 위 셋 + info

### 필터 (검색바)
- **레벨**: 보고 싶은 종류만 (error만, guard만 등)
- **source**: 함수명 검색 (예: `pollForUpdates`, `saveLS`, `sbSaveAll`)
- **기간**: 24시간 / 7일 / 30일 / 90일

### 목록 (시간 역순)
각 행 클릭 → 상세 모달:
- 정확한 발생 시각
- 어떤 회사·사용자에서 발생했는지
- 어느 화면(URL)에서
- 스택 트레이스 (개발자 분석용)
- 추가 메타 정보 (가드의 경우 어떤 키를 막았는지 등)

## 3. 신호별 대응 가이드

### 🛡️ guard 로그 발견 시 (가장 중요)

**guard 신호 = 데이터 유실 사고를 1차 가드가 막은 경우**

| meta.reason | 의미 | 권장 대응 |
|------------|------|---------|
| `snap_null` | 초기 로드 전 빈값 저장 시도 차단 | 1~2회는 정상 (페이지 새로고침 직후). 같은 사용자에게 반복 발생하면 점검 |
| `snap_has_data` | 데이터 있는 키가 빈값으로 덮어쓰기 시도 | **즉시 점검** — 1·2·3차 wipe 사고와 같은 패턴. 어떤 사용자·시각·키인지 확인 후 개발자 연락 |

상세 모달에서 `meta.key`로 어떤 데이터(emps/rec/bonus 등)인지 확인.

### 🟡 warn — pollForUpdates 504/timeout

- 1주에 5건 이하: 정상 (큰 회사 첫 폴링에서 가끔 발생)
- **1일에 20건 이상**: 데이터 크기가 임계점 도달. 개발자에게 sbLoadAll 청크 분할 도입 검토 요청
- **특정 회사 ID에 집중**: 그 회사가 너무 큰 상태. 개별 청크 분할 또는 rec 월별 분할 검토

### 🔴 error — window.onerror / unhandledrejection

- 같은 메시지가 5건 이상 누적: **즉시 개발자 연락**. 신규 버그 가능성 높음
- 1~2건: 일시적 (네트워크 끊김 등). 추세 관찰

### 🔴 error — saveLS

- `isQuota: true`: localStorage 한도 초과. 사용자 PC에 다른 데이터 누적. 그 사용자에게 캐시 정리 안내
- 일반 에러: 개발자 점검

---

## 4. 자주 보게 될 정상 패턴

다음은 **무시해도 되는** 정상 신호:

- `client:window.onerror` — `Script error.` (외부 광고·확장프로그램 노이즈)
- `pollForUpdates` warn 1~2건/일 — 일시 네트워크 불안정
- `safeItemSave` guard `snap_null` — 새로고침 직후 정상 동작

**비정상 신호 (즉시 점검):**
- 같은 메시지 10건 이상 누적
- error/guard가 평소 3배 이상
- 새로 등장한 source (없던 함수명에서 에러)

---

## 5. 개인정보 보호 — 자동 스크럽 동작

로그 메시지에 다음 정보가 포함되면 **자동 마스킹**됩니다:
- 주민번호 → `901231-*******`
- 사업자번호 → `***-**-*****`
- 전화번호 → `010-****-****`
- 이메일 → `s****n@naver.com`
- AES 암호화 데이터 → `ENC:[REDACTED]`
- 비밀번호·토큰·해시 등 키 → 값 자체 `[REDACTED]`

스크럽은 **클라이언트(전송 직전) + 서버(저장 직전)** 이중으로 적용됩니다.

이 로그를 외부에 노출(스크린샷 공유 등) 하더라도 개인정보 사고로 이어지지 않도록 설계되어 있지만, **스크린샷 공유 시 회사 ID·사용자 이메일은 보일 수 있으니** 주의하세요.

## 6. 데이터 보존

- **90일** 자동 보관 후 자동 삭제 (pg_cron 매일 새벽 3시)
- 그 이상 보관이 필요한 사고 데이터는 백업 따로 받아두세요 (관리자 SQL Editor에서 `error_log` 테이블 export)

---

## 7. 사이드바 뱃지 숫자

- 0 (뱃지 안 보임): 모든 정상
- 1~99: 미해결 error 누적 개수
- "99+": 100개 이상 → **즉시 점검 권장**

뱃지는 [📊 모니터링] 탭 진입 시 자동 갱신됩니다.

---

## 8. 개발자에게 전달 시

이상 징후 발견 시 개발자에게 전달할 정보:
1. 발생 시각 (정확한 시각)
2. source (어느 함수)
3. 메시지 (전체 텍스트)
4. 스택 트레이스 (상세 모달에서 복사)
5. 발생 빈도 (몇 명·몇 회사에서)
6. 회사 ID (특정 회사에 집중되는지)

상세 모달에서 모든 필드 복사 가능합니다.
